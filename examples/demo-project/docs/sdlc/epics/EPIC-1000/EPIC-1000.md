# EPIC-1000 — Bulk-assign cards from board view

## Goal

Let workspace members and admins multi-select cards on a board and assign them
to a single teammate in one action.

## Why

Sprint kickoff and triage today is repetitive: open card → assign → close →
next card, ×N. Power users have asked for this twice in the last quarter
(see support tickets TF-2421, TF-2509). Estimated time saved per triage
session: ~3 minutes for a 20-card sprint.

## In scope

- Multi-select on the board view (shift-click + ctrl/cmd-click).
- A single "Assign to…" action that applies to all selected cards.
- Assignment results in one notification per card (existing
  `card.assigned` notification, fan-out as today).
- Audit log entry per card.

## Out of scope

- Bulk *un*-assign (separate epic).
- Bulk move across columns or boards.
- Bulk label / due-date edit.
- Mobile / touch multi-select gesture.

## Affected modules

- `auth` — assignment authorization (only workspace members; not guests).
- `tasks` — board UI, multi-select state, `PATCH /cards/:id/assignees`.

## Dependencies

- None. Existing assignment endpoint is reused.

## Out-of-scope but related

- The assignment endpoint already handles 403 when the assignee is a
  guest — we keep that behavior, but the multi-select UI must surface
  per-card failures cleanly (see PRD AC-04).

## Risks

- Performance: 50-card bulk assign could dominate the assignment topic.
  Tech Lead will decide between client-side fan-out and a new bulk
  endpoint.
- UX clarity: when some cards fail (e.g. one is a guest's), users need
  to understand which.

## Success metrics

- ≥ 30 % of triage sessions use bulk-assign within 2 weeks of launch.
- No regression on assignment p95 latency.
- Zero assignment-permission incidents in the first month.
